- スポーツ施設、宿泊、サウナでシステムが分断（Googleカレンダーベース）
- 「施設×日付×時間」の掛け合わせが見づらい
- ノーショー（無断キャンセル）の発生
- スタッフのITリテラシーに差がある

## 在庫管理の設計思想: セット品番方式（塊管理）

ECのセット品番と同じ考え方で在庫を管理する。

| 概念 | EC用語 | 予約システム |
|---|---|---|
| 親SKU（在庫の実体） | 単品 | 物理部屋 × 1時間スロット |
| 子SKU（セット品番） | セット品番 | 予約プラン（宿泊=18スロット束、日帰り午前=4スロット束 等） |
| 在庫連動 | セットが売れたら単品在庫が減る | 日帰りで抑えたら宿泊できない |

### なぜこの設計か
- 同じ部屋（6畳）が宿泊でも日帰りでも使われる
- 日帰り通しで8:30-22:00を抑えたら、その日の宿泊(16:00〜翌10:00)は不可
- 1時間スロットが在庫の最小単位なので、どんなプランの組み合わせでも競合を正確に検出できる

### 宿泊の翌朝またがり
- 宿泊(IN 16:00/OUT 10:00) → 当日の[16-23]スロット + 翌日の[0-9]スロットを確保
- 2泊 = 42時間の確保（4/1 16:00 〜 4/3 10:00）
- `expandStaySlots()` で翌朝スロットを正しい日付に変換して空きチェック
